# Geometry of Reasoning Trajectories

Изучение геометрии латентных траекторий рассуждения в рекуррентно-глубинном
трансформере **Huginn-3.5B**. Проект **inference-only**: модель не обучается,
мы только прогоняем её и измеряем форму латентного пути.

---

## 1. Что это

Современные «reasoning»-модели обычно рассуждают *словами* — выписывают цепочку
токенов (chain-of-thought). Huginn устроен иначе: у него есть **единый
рекуррентный блок**, который прогоняется по одному и тому же скрытому состоянию
`r` раз подряд, прежде чем выдать ответ. Рассуждение происходит не в тексте, а в
**латентном пространстве** — «в уме» модели. Это и есть *latent reasoning*.

Каждый такой прогон порождает **траекторию** скрытого состояния одного токена:
последовательность `h_0, h_1, …, h_r` в пространстве размерности `hidden_dim`.
Нас интересует **геометрическая форма** этой траектории. Эмпирически она бывает
трёх типов:

- **сходится (settle)** — путь быстро затухает в неподвижную точку;
- **петляет (loop)** — путь возвращается близко к уже пройденной точке, наматывая
  витки;
- **дрейфует (drift)** — путь уходит вдаль, не сходясь и не замыкаясь.

Гипотеза в том, что форма траектории связана с тем, *как* и *насколько глубоко*
модель рассуждает над задачей. Мы проверяем это, ничего не обучая: берём готовую
модель, гоняем её на задачах разной глубины и меряем геометрию.

---

## 2. Гипотезы

- **H1 — форм мало.** Всё многообразие латентных траекторий укладывается в
  небольшое число качественных типов (сходится / петляет / дрейфует), а не в
  континуум ни на что не похожих кривых.
- **H2 — число оборотов петли растёт с глубиной рассуждения `d`.** У «петлящих»
  траекторий количество витков (winding number) монотонно связано с числом шагов
  рассуждения, которое требует задача. **← это наш MVP спринта.**
- **H3 — сжатие ⇒ нельзя держать счётчик ⇒ счётные задачи обязаны петлять.**
  Если рекуррентный блок сжимает информацию, он не может хранить растущий
  счётчик в неподвижной точке; значит, задачи со счётом вынуждены наматывать
  петли, а не сходиться.

> **Фокус спринта — H2.** H1 и H3 остаются на потом; вся инфраструктура
> строится так, чтобы сначала довести до конца измеримую H2.

---

## 3. Что меряем (4 метрики)

- **Показатель Ляпунова** — скорость расхождения близких траекторий (хаотичность/
  устойчивость динамики).
- **Спектральный радиус якобиана** — локальное сжатие/растяжение шага рекуррентного
  отображения (< 1 ⇒ сжатие ⇒ склонность сходиться).
- **Персистентная гомология H1** — топологический признак наличия «дырки», то есть
  настоящей петли в траектории.
- **Число оборотов (winding number)** — сколько витков наматывает 2D-проекция пути
  вокруг центра. **← основная метрика MVP.**

---

## 4. Формализация MVP (H2)

Конвейер для проверки H2:

1. **Прогон.** Гоним Huginn на задаче глубины `d`, снимаем траекторию токена
   ответа: `states` формы `[num_steps + 1, hidden_dim]`.
2. **Loop-gate.** Классификатор формы отбирает только **петли** — winding number
   осмыслен лишь на них (settle/drift отбрасываем).
3. **Проекция.** PCA-проекция траектории в 2D (`[T, 2]`).
4. **Winding number.** Считаем знаковое число оборотов 2D-пути вокруг центроида.
5. **Корреляция.** Спирмен `ρ(w, d)` + `p-value`: растёт ли число витков с
   глубиной.
6. **Контроль конфаундера длины `L`.** Длинные промпты могут сами по себе менять
   геометрию, поэтому длину `L` выносим за скобки — частная корреляция (pingouin)
   или вычет длины регрессией.
7. **Сиды.** Старт рекуррентного блока `h_0` случаен, поэтому **фиксируем сиды** и
   усредняем по ним — иначе результат невоспроизводим.

**Критерий «готово»:** воспроизводимая пара `(ρ, p)` минимум на двух разных
глубинах `d` + scatter-plot «winding vs depth».

## 5. Контракт формата

Весь код пишется против одного разделяемого dataclass. **Ключевое поле —
`states` формы `[шаги, размерность]`.** Благодаря этому синтетические пути
(окружность, спираль) и реальные `h_t` из модели **взаимозаменяемы**: gate,
winding и projection тестируются на синтетике без модели, а потом запускаются на
настоящих траекториях без единой правки.

```python
@dataclass(frozen=True)
class Trajectory:
    states: np.ndarray   # [num_steps + 1, hidden_dim] — путь h_0..h_r
    token_index: int     # позиция анализируемого токена (-1 = последний/ответ)
    seed: int            # сид случайного старта h_0 (воспроизводимость)
    depth: int           # d — глубина рассуждения задачи
    seq_len: int         # L — длина промпта (конфаундер, контролируем)
    prompt_id: str       # идентификатор исходного промпта

    def save(self, path: str) -> None: ...
    @classmethod
    def load(cls, path: str) -> "Trajectory": ...
```

Реализация — в [src/traj_geom/types.py](src/traj_geom/types.py) (единственный
полностью готовый файл).

## 8. Как работать

- Каждый пишет **только в своей папке** — так конфликтов в гите нет.
- Короткие ветки `feature/<модуль>` → PR в `main`.
- **Синтетика позволяет писать и тестировать gate/winding БЕЗ модели.** Генерируешь
  окружность (заведомо знаешь: оборотов = 1) и проверяешь, что твой код выдаёт
  именно это. То же для spiral_in (должно быть `settle`), line (`drift`) и т.д.
- Пиши против `Trajectory.states` `[шаги, размерность]` — тогда твой код без правок
  переедет с синтетики на реальные `h_t`.
- Тесты геометрии сейчас лежат под `@pytest.mark.skip` — снимаешь skip, как только
  реализовал соответствующую заглушку.
